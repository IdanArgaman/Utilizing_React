This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Purpose

This is a personal React learning/reference playground, not a product codebase. Each top-level folder under `src/` is an independent "mini app" demonstrating one React concept (hooks, HOCs, context, Redux, routing). They are not meant to interoperate — treat each folder as its own sandbox.

The most actively developed part of this repo is `src/react-gotchas/` — a growing collection of React interview-prep "gotcha" examples (stale closures, batching, race conditions, referential identity, memoization strategies, etc.), each with a broken version and a fixed version side by side, heavily commented to explain the *why*. **When the user says "add to gotcha" (or similar shorthand), they mean: add a new numbered example to `src/react-gotchas/`, following the existing pattern** (see Architecture below).

## Commands

- `yarn start` — run the Vite dev server (hot reload).
- `yarn build` — production build via Vite, output to `dist/`.
- `yarn preview` — preview the production build locally.

There is no configured lint or test runner. A couple of `.spec.js` files exist under `src/todos-redux-example/` but no test framework (jest/vitest) is installed and no `test` script is defined — don't assume they run. `eslint-config-prettier`/`eslint-plugin-prettier` are devDependencies but there is no `.eslintrc`, so there's no active lint command either.

The `README.md` in this repo is stale Create React App boilerplate left over from before the migration to Vite/Yarn 4 — do not treat it as authoritative (e.g. it references `yarn eject` and a `build` folder that no longer apply).

## Architecture

### Top-level routing (`src/App.jsx`)

`src/App.jsx` owns the single `BrowserRouter` for the entire app (there must only ever be one — nesting a second `BrowserRouter` anywhere creates a conflicting history object). It renders:
- A back-arrow icon button (visible on every route except `/`) that links to `/`.
- A `<Switch>` with one `<Route>` per entry in `src/appsList.js`, plus a `/` route rendering `src/containers/Home.jsx`.

`src/appsList.js` is the single source of truth for every example app: each entry has `path`, `label`, `description`, and `component`. Both the Home page's card list and `App.jsx`'s route table are generated by mapping over this array — adding a new top-level example app means adding one entry here, not touching the router or Home page directly.

This project uses **react-router-dom v5** (`Switch` + `Route component={...}`, not v6's `Routes` + `element={...}`). Match that API in any new routing code.

### Sub-app pattern

Every folder under `src/` other than `containers/` (e.g. `useEffect/`, `hoc/`, `useContext/`, `tracer-by-hoc/`, `error-handling-hoc/`, `todos-redux-example/`, `react-router/`, `react-gotchas/`) exports a default `App` component from its own `App.jsx`, self-contained with its own `containers/`/`components/` subfolders. None of these sub-apps render their own `BrowserRouter` — they rely on the one provided by the top-level `App.jsx`.

### `src/react-gotchas/` — the gotcha examples app

This is where new work is concentrated. Structure:
- `src/react-gotchas/App.jsx` exports an `examples` array (path/label/component) — the same generator pattern as `appsList.js`, but scoped to gotcha examples. Adding a new gotcha means adding one entry here.
- `src/react-gotchas/containers/Home.jsx` lists all entries from `examples` as links.
- Each gotcha lives in its own `src/react-gotchas/containers/<Name>/<Name>.jsx` folder.

**Established convention for every gotcha example** (follow this for any new one):
1. A large top-of-file comment block explaining the concept: the problem, why it happens, and the fix — written for someone learning the concept, not just a code comment.
2. A "Broken" version of the component demonstrating the bug live in the browser (not just in a code comment) — prefer on-screen state/logs over `console.log` alone, since the user reads through the running app, not devtools.
3. A "Fixed" version (sometimes multiple fixing strategies) rendered side by side with the broken one for direct comparison.
4. Simulated async work (fake fetch with randomized delay, `setTimeout`, etc.) instead of a real backend, so race conditions and timing bugs are actually reproducible by interacting with the page.
5. Render counters (via a `useRenderCount` `useRef` helper) to make invisible re-render behavior visible on screen when the gotcha is about re-renders/memoization.

Numbering in `label` strings (e.g. `"9. Referential Identity..."`) is sequential across the whole `examples` array and should continue incrementing for new entries.
